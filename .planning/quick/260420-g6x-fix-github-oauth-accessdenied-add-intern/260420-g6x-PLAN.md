<objective>
Fix GitHub OAuth AccessDenied error caused by the signIn callback using localhost:5000
inside the Docker container instead of the api service hostname.

Purpose: The signIn callback runs server-side inside nodefy-frontend. Calling
localhost:5000 hits the container's own loopback (nothing listening), the fetch fails
silently, signIn returns false, and Auth.js emits AccessDenied. Adding INTERNAL_API_URL
pointing to http://api:5000 routes the call correctly within the Docker network.

Output:
- frontend/auth.ts updated to use INTERNAL_API_URL with fallback chain
- docker-compose.yml updated with INTERNAL_API_URL in the frontend service environment
</objective>

In the `frontend` service `environment` block (currently lines 43-51), add a new entry
immediately after NEXT_PUBLIC_API_URL:

```yaml
INTERNAL_API_URL: http://api:5000
```

`api` is the Docker Compose service name for the backend (container_name: nodefy-api).
Within the Docker bridge network, service names resolve as hostnames, so
`http://api:5000` is the correct internal URL.

Do NOT change NEXT_PUBLIC_API_URL — it must remain http://localhost:5000 (or the host
value) because it is also used in browser-side fetches that run outside the container.

In the signIn callback (line 15), replace:

```typescript
const apiUrl = process.env.NEXT_PUBLIC_API_URL ?? "http://localhost:5000"
```

```typescript

  process.env.INTERNAL_API_URL ??
  process.env.NEXT_PUBLIC_API_URL ??
```

Fallback chain rationale:
1. INTERNAL_API_URL — set in Docker; routes to http://api:5000 within the network
2. NEXT_PUBLIC_API_URL — set outside Docker or in .env.local for local dev
3. "http://localhost:5000" — last-resort default for bare local dev with no env file

No other changes to auth.ts. The rest of the signIn callback (mintApiToken, fetch body,
dto.id assignment) remains exactly as-is.
