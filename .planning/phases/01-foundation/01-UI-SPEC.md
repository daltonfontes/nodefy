# Phase 1 — UI Design Contract: Foundation

> Visual and interaction contract for Phase 1: SSO login, workspace creation, member invitation flow, and member list management.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

---

| Property | Value |

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding |
| sm | 8px | Button internal gap (icon + label), form field helper text offset |
| md | 16px | Default element spacing, card padding, input padding |
| lg | 24px | Section spacing within a card, form group separation |
| xl | 32px | Between major sections on page |
| 2xl | 48px | Vertical padding of auth card from viewport center |
| 3xl | 64px | Page-level top/bottom padding on centered layouts |

Exceptions:
- SSO provider buttons: min-height 44px (touch target minimum per WCAG 2.5.5)
- Workspace name input: height 40px (shadcn default `h-10`)
- Member table row height: 56px for comfortable tap targets on mobile

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 (regular) | 1.5 |
| Label | 14px | 500 (medium) | 1.4 |
| Heading | 20px | 600 (semibold) | 1.2 |
| Display | 28px | 700 (bold) | 1.15 |

Notes:
- Display is used only on the login page hero heading ("Nodefy") and the workspace creation page title.
- Heading is used for card titles (e.g., "Create your workspace", "Team members").
- Label is used for form labels, table column headers, and role badges.
- Body is used for all other text: descriptions, input values, helper text, empty state bodies.
- Font stack: `'Inter', ui-sans-serif, system-ui, -apple-system, sans-serif`

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `hsl(0 0% 100%)` — white | Page background, form inputs, auth card background |
| Secondary (30%) | `hsl(210 40% 96.1%)` — slate-100 | App shell background (behind centered cards), sidebar in future phases, table row striping |
| Accent (10%) | `hsl(221.2 83.2% 53.3%)` — blue-600 | Primary buttons, active state, focus ring |
| Destructive | `hsl(0 72.2% 50.6%)` — red-600 | Remove member button, destructive dialog confirm button |

Accent reserved for:
1. Primary CTA buttons (Continue with GitHub, Continue with Google, Continue with Microsoft, Create workspace, Send invite)
2. Focus ring on all interactive form elements (2px solid, 2px offset)
3. Active/selected role badge outline

Dark mode: not in scope for Phase 1. Tailwind `dark:` classes should not be added in this phase.

**Semantic color additions:**
- Success: `hsl(142.1 76.2% 36.3%)` — green-600 — used only for invite accepted confirmation badge
- Muted text: `hsl(215.4 16.3% 46.9%)` — slate-500 — used for helper text, placeholder text, secondary labels

## Screen Inventory

Phase 1 contains exactly 5 screens/flows:

### Screen 1: Login Page (`/login` or `/`)

**Layout:** Full-viewport centered column. No nav. No sidebar.

**Elements:**
- Nodefy wordmark (Display/28px/700) + optional small tagline (Body/14px/400, muted)
- Tagline text: "CRM para times de vendas" (muted, Body)
- Three SSO buttons stacked vertically, each full-width within a 360px max-width card:
  - "Continuar com GitHub" (GitHub icon, 20px, left-aligned icon)
  - "Continuar com Google" (Google icon, 20px, left-aligned icon)
  - "Continuar com Microsoft" (Microsoft icon, 20px, left-aligned icon)
- Each button: height 44px, border variant (not filled), full-width, icon + label
- Separator with "ou" between provider groups is not needed — all three are equal
- Footer: "Ao entrar, você concorda com os Termos de Uso e Política de Privacidade" (12px, muted, links underlined)

**States:**
- Default: all three buttons enabled
- Loading: clicked button shows spinner (animate-spin, 16px lucide `Loader2`) replacing icon; other buttons disabled with `opacity-50`
- Error: Alert component below buttons — destructive variant — "Não foi possível autenticar. Tente novamente."

**Card dimensions:** max-width 360px, centered horizontally and vertically, padding 32px (xl), border `border border-slate-200 rounded-xl shadow-sm`

---

### Screen 2: Workspace Creation (`/workspace/new`)

**Layout:** Full-viewport centered column. No nav. Back link not needed (no previous context on first login).

**Elements:**
- Heading: "Crie seu workspace" (Heading/20px/600)
- Body: "Seu workspace é onde seu time gerencia pipelines e deals." (Body/14px/400, muted)
- Form field: Workspace name
  - Label: "Nome do workspace" (Label/14px/500)
  - Input: placeholder "Ex: Vendas — Acme Corp", max 50 chars, `h-10`
  - Helper text below input: "Visível para todos os membros do workspace." (12px, muted)
- Primary CTA button: "Criar workspace" (full-width, accent background, white text)
- Loading state: button shows `Loader2` spinner + "Criando..." text, disabled

**Validation:**
- Required: inline error below input "Nome é obrigatório" (red-600, 12px) on blur if empty
- Min 2 chars: "Nome deve ter pelo menos 2 caracteres"
- Max 50 chars: character counter visible at `h-10` right-aligned in input when > 40 chars

**Card dimensions:** same as Login — max-width 360px, centered, padding 32px

---

### Screen 3: Member Invite Flow (`/workspace/[id]/settings/members/invite`)

**Layout:** Settings layout — full-page with a top nav bar showing workspace name + user avatar. Content area centered, max-width 640px.

**Elements:**
- Heading: "Convidar membro" (Heading/20px/600)
- Form:
  - Email field: Label "Email", Input type="email", placeholder "nome@empresa.com"
  - Role select: Label "Papel", shadcn Select component with two options:
    - "Membro" (default) — can view and interact with pipelines and cards
    - "Admin" — can manage pipelines, stages, and members
  - Role helper: "Admins podem gerenciar membros e pipelines." (12px, muted)
- Primary CTA: "Enviar convite" (full-width within form, accent)
- Cancel link: "Cancelar" — text link, no background, muted — navigates back to member list

**States:**
- Idle: form enabled
- Loading: "Enviando..." with spinner, form disabled
- Success: inline success alert — "Convite enviado para nome@empresa.com." — green-600 icon + text. Form resets for another invite.
- Error — duplicate: "Este email já é membro ou tem um convite pendente."
- Error — invalid email: "Email inválido." (inline below input)
- Error — network: destructive Alert above form — "Erro ao enviar convite. Tente novamente."

**Open question (STATE.md):** Invite delivery (email SMTP vs. in-app link) is unresolved. The UI spec is delivery-agnostic — the form collects email + role regardless. If shareable link is chosen, the success state should additionally show the generated invite link in a copy-to-clipboard input field.

---

### Screen 4: Invite Accept Landing (`/invite/[token]`)

**Layout:** Full-viewport centered column. No nav. Token is validated server-side.

**States:**

**Valid token — user not logged in:**
- Heading: "Você foi convidado para [Workspace Name]" (Heading/20px/600)
- Body: "Faça login para aceitar o convite e começar a colaborar." (Body/14px/400)
- Same three SSO buttons as Login page (identical visual treatment)
- After SSO: auto-redirect to workspace

**Valid token — user already logged in:**
- Heading: "Aceitar convite para [Workspace Name]" (Heading/20px/600)
- Body: "Você será adicionado como [role] neste workspace." (Body/14px/400)
- Accept button: "Aceitar convite" (primary, accent)
- Decline link: "Recusar" (muted text link)

**Invalid/expired token:**
- Heading: "Link inválido ou expirado" (Heading/20px/600)
- Body: "Este convite não é mais válido. Peça ao admin do workspace um novo convite." (Body/14px/400)
- No action buttons — dead end with informational content only

---

### Screen 5: Member List (`/workspace/[id]/settings/members`)

**Layout:** Settings layout — top nav + content area max-width 768px.

**Elements:**
- Page heading: "Membros" (Heading/20px/600)
- Subheading: "[N] membros" (Body/14px/400, muted) — live count
- "Convidar membro" button — top-right, accent, `h-9` (compact)
- Member table (shadcn Table component):
  - Columns: Avatar + Name + Email | Role | Joined | Actions
  - Avatar: 32px circle, initials fallback if no photo
  - Role: Badge component — "Admin" (blue outline) / "Membro" (slate outline)
  - Actions (Admin-only): dropdown menu (lucide `MoreHorizontal` trigger):
    - "Tornar admin" / "Tornar membro" — role toggle
    - Separator
    - "Remover do workspace" — destructive, red text

**Pending invites section** (below active members):
- Section label: "Convites pendentes" (Label/14px/500)
- Same table structure but with "Pendente" badge (amber outline) and "Reenviar" / "Cancelar" actions
- Empty: section not shown if no pending invites

**Empty state (no members — impossible in practice since creator is always member 1, but guard against it):**
- Center-aligned: "Nenhum membro ainda" heading, "Convide alguém para começar." body, "Convidar membro" button

**Destructive action — Remove member:**
- shadcn Dialog (not browser confirm)
- Dialog heading: "Remover membro"
- Dialog body: "Tem certeza que deseja remover [Name] do workspace? Eles perderão acesso imediatamente."
- Buttons: "Cancelar" (outline) + "Remover" (destructive variant, red background)

**Destructive action — Cancel invite:**
- Inline confirmation via Alert Dialog (not full Dialog)
- "Cancelar convite para [email]?" — "Sim, cancelar" (destructive) + "Não" (outline)

## Interaction Contracts

### SSO Button Loading State

When a provider button is clicked:
1. Replace the provider icon with `<Loader2 className="animate-spin h-4 w-4" />`
2. Set `disabled` on all three buttons
3. Button text changes: "Continuar com GitHub" → "Conectando..." (same button)
4. On error: restore all buttons, show Alert below button group
5. Timeout: if no response in 15s, restore buttons + show error

### Form Validation Timing

- Validate on `blur` (not `onChange`) for all text inputs to avoid noisy inline errors
- Validate on submit for completeness
- Show character counter only when input > 80% of max (e.g., > 40 chars for 50 char max)

### Role Change (Member List)

1. Optimistic: Badge updates immediately in the table row
2. If API call fails: badge reverts + toast: "Não foi possível alterar o papel. Tente novamente."
3. Admin cannot demote themselves — "Tornar membro" is hidden for the current user's own row

### Remove Member (Member List)

1. Click "Remover do workspace" → Dialog opens
2. Click "Remover" → row fades out optimistically (opacity-50, pointer-events-none)
3. On API success: row removed from list
4. On API failure: row restores opacity + toast: "Não foi possível remover o membro."

## Responsive Behavior

Phase 1 screens are centered layouts — responsive rules are minimal:

| Breakpoint | Behavior |
|------------|----------|
| < 640px (mobile) | Auth card fills full width with 16px horizontal padding; no card border/shadow |
| 640px–1024px (tablet) | Auth card max-width 360px centered; Settings content max-width 100% with 24px padding |
| > 1024px (desktop) | Auth card max-width 360px; Settings content max-width 768px |

Member table on mobile: collapse to card list layout (Name + Role + Actions stacked). Email column hidden below 640px.
